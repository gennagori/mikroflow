# MikroTik NetFlow Collector — дизайн

Дата: 2026-07-01
Статус: утверждён (дизайн)

## 1. Цель

Собирать с роутера MikroTik информацию о сетевых соединениях («кто и куда
подключался») для последующего анализа и хранить её 6 месяцев в PostgreSQL.

Для каждого соединения нужно знать:
- **src_ip** — кто инициировал (внутреннее устройство);
- **имя устройства** — hostname из аренд DHCP;
- **dst_ip** — куда;
- **домен назначения** — reverse-DNS от dst_ip;
- объём (байты/пакеты), порт/протокол, время.

## 2. Требования и ограничения

- Источник данных — **NetFlow v9/IPFIX** (MikroTik Traffic Flow). Лёгкий путь,
  без зеркалирования трафика.
- **DNS клиентов менять нельзя** → домен получаем reverse-DNS'ом от dst_ip
  (с кэшем). Известное ограничение: домены CDN определяются неточно.
- Имя устройства — из **RouterOS API** (`/ip/dhcp-server/lease`).
- **Хранение — гибрид:** сырые флоу ~14 дней, почасовые агрегаты 6 месяцев.
- Масштаб — **100+ устройств** → партиционирование обязательно.
- Развёртывание — **Docker Compose на Linux**, БД — PostgreSQL.

## 3. Архитектура

Четыре небольших независимых сервиса + Postgres. Каждый сервис имеет одну
ответственность, общается через таблицы БД.

```
 MikroTik ──NetFlow(UDP 2055)──▶ [collector] ──▶ flows_raw (партиции по дням)
 MikroTik ──RouterOS API───────▶ [lease-sync] ──▶ dhcp_leases (IP→имя, история)
 flows_raw (новые dst-IP) ─────▶ [rdns-worker] ─▶ ip_domain (кэш IP→домен)
 по расписанию ────────────────▶ [aggregator] ─▶ flows_hourly (партиции по мес.)
                                        └─ ретеншн: дроп старых партиций
                                             PostgreSQL
```

### Компоненты

- **collector** — слушает UDP 2055, парсит NetFlow v9/IPFIX (шаблоны кешируются
  per-exporter), пакетно пишет сырые флоу в `flows_raw`. Библиотека парсинга
  NetFlow для Python (v5/v9/IPFIX).
  - **Развязка приёма и записи:** UDP-приёмник только парсит и кладёт записи в
    **bounded-очередь в процессе**; отдельный поток батчами пишет в Postgres
    через `COPY`. Плюс увеличенный `SO_RCVBUF`. Это защищает от потери UDP-пакетов
    при подвисании БД **без отдельного брокера**.
  - Путь записи спрятан за интерфейс `FlowSink` — чтобы при будущем росте
    (тысячи устройств / мульти-хост) можно было подменить на Redis/Kafka без
    переписывания коллектора.
  - **Почему не Redis сейчас:** «большой объём» здесь — это 6-месячный архив на
    диске (решается партиционированием), а не пропускная способность приёма.
    Для 100+ устройств (сотни–тысячи flow/сек) один процесс с батчевым `COPY`
    справляется с запасом; отдельный брокер — преждевременное усложнение (YAGNI).
- **lease-sync** — раз в N минут опрашивает `/ip/dhcp-server/lease` по API,
  ведёт `dhcp_leases` с историей (`valid_from/valid_to`), т.к. DHCP
  переназначает адреса.
- **rdns-worker** — находит dst_ip без домена, делает PTR-запрос, кэширует
  (TTL; NXDOMAIN тоже кэшируется, чтобы не повторять запросы).
- **aggregator** — раз в час сворачивает `flows_raw` в `flows_hourly`,
  **денормализуя** имя устройства (по аренде, актуальной на тот час) и домен
  прямо в строку — чтобы 6-месячные данные оставались самодостаточными даже
  после ротации аренд/PTR.

## 4. Модель данных

```sql
-- Сырые флоу. Партиции по дням, ретеншн ~14 дней.
flows_raw (
  ts, src_ip, dst_ip, src_port, dst_port, protocol,
  bytes, packets, exporter_ip
)

-- Аренды DHCP с историей.
dhcp_leases (
  ip, mac, hostname, valid_from, valid_to
)

-- Кэш reverse-DNS.
ip_domain (
  ip, domain, resolved_at, ttl, status  -- status: ok | nxdomain
)

-- ГЛАВНАЯ таблица анализа. Партиции по месяцам, ретеншн 6 мес.
flows_hourly (
  hour, src_ip, device_name, dst_ip, dst_domain,
  dst_port, protocol, bytes, packets, flow_count
)
```

Финальная вьюха `v_connections` поверх `flows_hourly`: «имя устройства,
src_ip → домен (dst_ip:port), объём, число соединений, время».

## 5. Ретеншн и объём

- `flows_raw` — партиции по дням, авто-дроп старше 14 дней (настраивается).
- `flows_hourly` — партиции по месяцам, авто-дроп старше 6 месяцев.
- Дроп партиций вместо `DELETE` — мгновенно и без раздувания таблиц.
- Прикидка `flows_hourly`: ~100 устройств × ~200 dst/час × 24 × 180 дн ≈
  десятки млн строк — Postgres тянет с партициями и индексами.
- Все пороги ретеншна — в конфиге, подстраиваются под реальный трафик.

## 6. Развёртывание

- Один `docker-compose.yml` — сервисы: `postgres`, `collector`, `worker`
  (lease-sync + rdns + aggregator + ретеншн внутри одного планировщика).
- Конфигурация через `.env`: IP роутера, логин/пароль RouterOS API, порт
  NetFlow, пороги ретеншна, интервалы опроса.

### Настройка на стороне MikroTik

```
/ip traffic-flow set enabled=yes interfaces=all
/ip traffic-flow target add address=<IP_сервера>:2055 version=9
/ip service enable api
```

## 7. Обработка ошибок

- **collector**: при недоступности БД буферизует записи в ограниченной очереди;
  NetFlow v9 data-flowset без пришедшего шаблона откладывается до получения
  шаблона.
- **lease-sync / rdns-worker**: сбои API и PTR логируются, ретрай с backoff.
- **rdns**: таймауты и NXDOMAIN кэшируются, чтобы не повторять запросы.

## 8. Тестирование

- Юнит-тесты: парсинг NetFlow (готовые байты-сэмплы v9/IPFIX), парсинг аренд
  DHCP, SQL агрегации, SQL ретеншна (дроп партиций).
- Интеграционный тест: фейковый отправитель NetFlow → collector → проверка
  записей в `flows_raw`; прогон агрегатора → проверка `flows_hourly`.

## 9. Вне рамок (YAGNI)

- Веб-интерфейс / дашборды (данные лежат в Postgres, анализ — SQL/BI-инструмент).
- Пассивный разбор DNS через зеркалирование трафика (выбран лёгкий путь).
- Логин пользователя (Hotspot/PPP) как имя — только hostname из DHCP.
- Внешний брокер (Redis/Kafka) — путь записи за интерфейсом `FlowSink`, добавим
  при реальном росте нагрузки.
